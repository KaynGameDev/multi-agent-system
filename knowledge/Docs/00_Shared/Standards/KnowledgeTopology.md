# Knowledge Topology

## 目的

这份文档用于约束 KB V1 中知识应该落在哪一层，避免不同层级混写，避免把历史材料误当成当前事实。

KB V1 只覆盖设计知识，Canonical 格式统一为 Markdown，不引入 ADR。

## 四层定义

### Shared

适合放：

- 公司级术语
- 设计原则
- 命名规范
- 文档规范
- 跨项目通用规则

不适合放：

- 某个大厅包体的当前规则
- 某个具体分支的当前数值
- 只在单个游戏或单个功能中成立的事实

### GameLine

适合放：

- 某个同源游戏家族的母体共性
- 共享玩法、共享系统、共享术语
- 各 Deployment 之间的差异维度说明

不适合放：

- 某个大厅分支的当前配置
- 某次活动版本的临时事实

### Deployment

适合放：

- 某个大厅 / 地区 / 包体下某个游戏分支的当前事实
- 分支特有功能、限制、规则、运营点
- 相对 GameLine 的明确差异

不适合放：

- 仍未确认的历史说法
- 其他大厅分支的事实

### Legacy

适合放：

- 旧文档
- 迁移中间产物
- 历史参考材料
- 尚未完成确认和升格的遗留资料

注意：

- `Legacy` 默认只是参考，不自动视为 Source of Truth
- `Legacy` 内容只能在被人工确认后，才能被提炼进其他层

## 放置规则

### 规则 1：先判断适用范围，再决定落层

- 全公司通用 -> 优先看 `Shared`
- 同源游戏家族共性 -> 优先看 `GameLine`
- 某个大厅 / 包体下的当前事实 -> 优先看 `Deployment`
- 历史资料或待确认材料 -> 优先看 `Legacy`

### 规则 2：同名游戏跨大厅不能合并为一个事实源

即使游戏名相同，只要落在不同大厅、地区或包体下，就可能已经分化出不同的分支事实。

因此：

- 同名不等于同一事实
- 同源不等于同一文档
- 共享 code base 不等于共享当前规则

### 规则 3：GameLine 和 Deployment 必须显式拆开

讨论某条知识时，要先问清楚：

- 这是母体共性，还是分支特例
- 这是当前事实，还是历史遗留
- 这是正式规则，还是曾经讨论过的方案

### 规则 4：Legacy 不能自动升格

如果某条内容只存在于旧文档、截图、会议纪要或迁移稿中：

- 可以作为线索
- 不可以直接当作已确认事实
- 必须经过人工确认，才能被沉淀进 `Shared`、`GameLine` 或 `Deployment`

## 常见误区

### 误区 1：把 Shared 写成“大而全的杂项层”

`Shared` 只放真正通用的知识，不是所有暂时不知道放哪的内容。

### 误区 2：把 Deployment 当前事实写成 GameLine 共性

如果只在单个大厅或单个包体中成立，就不应写进 `GameLine`。

### 误区 3：把 Legacy 当作当前版本说明

旧文档可以参考，但不能默认代表当前版本。

## 文档落点建议

- 公司术语表、命名规范、设计规范 -> `Shared`
- 游戏母体概览、共享系统、共享术语 -> `GameLine`
- 分支版 `master_gdd`、分支特有 `feature_spec` -> `Deployment`
- 未审查的旧文档、迁移草稿、历史材料 -> `Legacy`
