# Skill: Elicit Feature Spec

## 目的

用于围绕单个功能或子系统，逐步抽取知识并沉淀成 `Feature Spec` 草稿。

---

## 适用场景

当用户要梳理：

- 一个功能
- 一个子系统
- 一个玩法模块
- 一个具体流程

例如：
- Combat
- Reward Flow
- Shop
- Login Flow
- New Player Guide

---

## 你的任务

### 1. 锁定本轮 Feature 范围
先明确：
- 功能名称
- 属于哪个 Deployment 或 GameLine
- 是母体共性还是分支特例

### 2. 按固定顺序提问
建议按以下顺序逐步提问：

1. 这个功能的目标是什么？
2. 它给玩家带来的价值是什么？
3. Scope / Non-scope 是什么？
4. 主流程是什么？
5. 核心规则是什么？
6. 有哪些状态变化？
7. 参数 / 公式 / 数据依赖是什么？
8. 边界情况 / 异常情况是什么？
9. 它依赖哪些其他系统？
10. 最近有哪些关键变更与决策？

### 3. 每轮总结
每轮都要整理成 Feature Spec 的结构化内容。

### 4. 主动查漏补缺
重点检查是否缺少：
- Scope / Non-scope
- 边界情况
- 依赖
- 决策记录
- 与其他分支的差异说明

---

## 输出格式

### 本轮目标
### 本轮问题
### 已确认内容
### 待确认内容
### 缺失章节提醒
### 当前可沉淀的 Feature Spec 骨架

---

## 注意事项

- 不要直接写成一大段说明文
- 优先围绕模板结构抽取
- Feature Spec 必须能承载“变更与决策记录”
- 遇到分支差异时，必须明确是否只适用于当前 Deployment
